## 1. 引言

### 1.1 项目背景与动机

当前主流 AI Agent 框架（LangChain、CrewAI、AutoGen 等）普遍面临一个核心瓶颈：**记忆能力的缺失或碎片化**。大多数 Agent 的"记忆"仅依赖 LLM 的上下文窗口（Context Window），这导致了三个根本性问题：

1. **上下文截断**：长对话和复杂任务链超出窗口限制后，关键信息丢失
2. **经验无法积累**：每次会话从零开始，无法从历史成功/失败中学习
3. **知识孤岛**：静态文档、实时交互、执行追踪等数据源之间缺乏关联

业界已开始探索解决方案。**Letta（原 MemGPT）** 率先提出了有状态 Agent 的概念，通过分层记忆（Core Memory / Archival Memory / Recall Memory）实现了跨会话的持久化记忆，使 Agent 能够随时间学习和自我改进 [2]。**LightRAG** 则在检索增强生成领域引入了图增强的双层检索架构（低层级实体检索 + 高层级主题检索），显著提升了多跳推理的准确性 [3]。**GraphRAG（微软）** 进一步将社区检测算法引入知识图谱，通过层级化的社区摘要实现了全局性问题的回答能力 [1]。

然而，这些方案各有局限：Letta 的记忆模型偏向对话场景，缺乏对企业级知识工程的支持；LightRAG/GraphRAG 聚焦于检索管道，缺乏完整的记忆生命周期管理。**本项目（AMS）的目标是构建一个生产级的、认知科学驱动的完整记忆基础设施**，融合上述前沿思想的优势，同时解决企业级部署的可靠性、可观测性和可扩展性需求。

### 1.2 设计目标

| **目标维度** | **具体指标** |
|---|---|
| **认知完备性** | 实现四种记忆类型（Working / Procedural / Declarative / Metacognitive）的完整覆盖 |
| **双轨存储** | 每种长期记忆同时支持 Tree（层级抽象）+ Graph（关系网络）双重结构 |
| **闭环飞轮** | 实现 Trace → Memory → Skill → Trace 的持续优化循环 |
| **检索性能** | Working Memory < 50ms p99；Long-term Memory < 200ms p95 |
| **生产就绪** | Kubernetes 原生部署、全链路可观测、故障自愈、水平扩展 |
| **开放集成** | 标准 REST/gRPC API、支持主流 Agent 框架（LangChain/LangGraph/CrewAI）接入 |

### 1.3 设计原则

| **原则** | **说明** |
|---|---|
| **搜索即核心** | 记忆的构建是为了支持有效的搜索。所有存储结构的设计决策都以检索效率为第一优先级（源自图2的核心设计哲学） |
| **认知科学驱动** | 记忆分类、遗忘机制、巩固过程均参照人类认知心理学模型 |
| **数据湖思维** | 采用 Memory Lake + Multimodal Lake 的分层存储架构，实现"AI Agent 时代的 ETL 和 DataLake"（源自图2） |
| **非侵入式采集** | 通过 Sidecar/eBPF/字节码插桩实现零代码改动的遥测数据采集 |
| **渐进式增强** | 系统可从单一 Working Memory 起步，逐步开启 Tree/Graph/Skill 等高级能力 |

## 2. 系统上下文与服务拓扑

### 2.1 认知闭环模型（The Cognitive Loop）

AMS 的核心架构建立在一个受认知科学启发的**三角闭环反馈模型**之上。如图1所示，**Agent Framework** 处于中心位置，围绕它有三大外围子系统通过不同的数据通道形成"飞轮效应"。

这一设计与 Letta 的"stateful agent"理念一脉相承——Agent 不再是无状态的请求-响应函数，而是拥有持久记忆、能够随时间学习和自我改进的有状态实体 [2]。

#### 三大核心子系统

| **子系统** | **全称** | **认知隐喻** | **核心职责** |
|---|---|---|---|
| **Agent-TES** | Agent Telemetry & Evidence Service | 感官输入 + 执行日志 | 非侵入式捕获 Agent 推理链、工具调用、环境反馈的细粒度遥测数据 |
| **Agentic Memory** | Agent Memory System | 知识库 + 经验库 | 将瞬时 Trace 转化为结构化的长期洞察，提供多模态检索 |
| **Skill-MDS** | Skill Mining & Discovery Service | 动作库 + 技能库 | 从成功的 Memory 模式中提炼可复用的技能配方 |

> ⚠️ **命名演进说明**：在早期 HLD 初稿中，Skill 子系统曾命名为 "Skill-DOM"（Discovery & Observation & Mining）。在版本B中统一更名为 **Skill-MDS**（Mining & Discovery Service），功能描述也从 4 项精简为 3 项（a. 技能目录/索引/检索/推荐；b. 技能网格/关系发现/拓扑映射；c. 技能挖掘/归纳/精炼），更加聚焦。

#### 飞轮闭环数据流

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   Agent Framework ──tracing──→ Agent-TES                │
│        ↑                           │                    │
│        │                    streaming data pipeline      │
│     applying                  (在线实时更新)              │
│        │                           │                    │
│        │                           ↓                    │
│   Skill-MDS ←──skill gen──── Agentic Memory             │
│                  pipeline      ↑    │                   │
│                             memory IO                   │
│                                │                        │
│                         Agent Framework                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**闭环运转逻辑**：

1. **Trace → Memory**（在线实时）：Agent-TES 通过 streaming data pipeline 将执行遥测数据实时注入 Agentic Memory，从噪声中提炼经验
2. **Memory → Skill**（离线挖掘）：Agentic Memory 通过 skill gen pipeline 将反复成功的记忆模式抽象为可复用的 Skill
3. **Skill → Agent**（运行时应用）：Skill-MDS 将技能应用回 Agent Framework，增强 Agent 的专业能力，产生新的 Trace，重启循环

### 2.2 Agent-TES 详细能力

Agent-TES 是整个系统的**数据采集层**，采用非侵入式架构，确保对 Agent 执行的零干扰。

| **能力域** | **具体内容** |
|---|---|
| **a. 非侵入式采集** | eBPF / Sidecar / Bytecode Instrumentation 三种采集模式，按环境灵活选择 |
| **b. 端到端活动记录与证据摄入** | • 用户请求、事件、上下文<br/>• 多 Agent 编排与跨 Agent 协调<br/>• LLM CoT 推理链、决策检查点<br/>• 模型版本、Token 消耗、决策置信度<br/>• 工具调用、Prompts、日志与证据<br/>• 结果与反馈 |
| **c. 归并入记忆系统** | 通过 streaming data pipeline 实时更新 + 离线批量归并 |

**技术实现要点**（源自 06-Agent-TES.md）：

- 基于 **OpenTelemetry** 协议标准化 Trace/Span 数据模型
- **Kafka** 作为 Trace 数据的缓冲层和持久化传输通道
- Sidecar 模式通过 Kubernetes Admission Webhook 自动注入
- 支持采样率动态调整（生产环境默认 10%，调试模式 100%）

### 2.3 Skill-MDS 详细能力

| **能力域** | **具体内容** |
|---|---|
| **a. 技能目录与检索** | Skill catalog（技能目录）、indexing（索引）、retrieval（检索）、recommendation（推荐） |
| **b. 技能网格与关系发现** | Skill mesh（技能网格）、relationship discovery（关系发现）、topology mapping（拓扑映射） |
| **c. 技能挖掘与精炼** | Skill mining（挖掘）、induction（归纳）、refinement（精炼）→ 最终输出 skill.md |

**技能生成全流程**（源自 05-Skill-Mining-and-Discovery.md）：

```
Pattern Detection → Candidate Extraction → Sandbox Testing → Skill Codification → skill.md
```

### 2.4 数据流与存储分层架构

图2展示了整个系统的**宏观数据流与存储分层**，这是理解 AMS 整体架构的关键视图。

#### 四层架构

| **层级** | **名称** | **内容** | **定位** |
|---|---|---|---|
| **L1 — 应用层** | Multi-Agent System | 业务系统/服务（API Gateway, MCP）+ 多 Agent 编排 | 消费者：通过 structured memory retrieval 获取记忆 |
| **L2 — 记忆层** | Memory Lake | Knowledge Tree, Graph, Time Series, Vector... | 核心：结构化记忆存储与检索引擎 |
| **L3 — 数据层** | Multimodal Lake | CoT, Docs, Video, Img, Table | 基座：AI Agent 时代的 ETL 和 DataLake |
| **L4 — 工程层** | Knowledge Engineering | Agentic Memory, Skill Gen, Agentic RL Dataset, Chain of Evidence | 引擎：将数据转化为知识的处理管线 |

#### 核心设计哲学

图2右侧标注了两条关键设计哲学：

> **"搜索才是记忆的核心，记忆的构建是为了支持有效的搜索"**

这意味着所有存储结构（Tree / Graph / Vector / TimeSeries）的设计决策都以检索效率为第一优先级。记忆的"写入"服务于"读取"。

> **"AI Agent 时代的 ETL 和 DataLake，将数据转化为知识，闭环服务于 Agent 系统"**

Multimodal Lake 不是传统的数据湖，而是面向 Agent 认知需求的多模态知识转化引擎。

#### 数据流向

**输入侧（左侧）：**
- **Trace System**：思考过程「CoT」、交互数据、引用证据、决策链路 → Memory Lake + Multimodal Lake
- **业务数据和信息**：business documents → Multimodal Lake

**输出侧（右侧）：**
- **Structured Memory Retrieval**：语义检索、精准匹配、粗召回、精排、结构化拼装 → Multi-Agent System

## 3. 记忆分类体系

### 3.1 认知科学基础

AMS 的记忆分类直接映射人类认知心理学的经典模型。以下是完整的概念映射表（源自补充表格1）：

| **维度** | **记忆类型** | **归属/等价关系** | **在 Agent 中的定义与实现** |
|---|---|---|---|
| 基础类别 | **Procedural Memory**（程序性记忆） | 与 Declarative Memory 相互独立 | **技能型记忆**。在 Agent 中表现为通过强化学习（RL）获得的策略或确定性硬编码逻辑 |
| 高层概念 | **Declarative Memory**（陈述性记忆） | 等价于 Factual Memory | **事实型记忆**。可以被显式表达、检索和描述的知识 |
| 陈述性子类型 | **Semantic Memory**（语义记忆） | Declarative Memory 的子集 | **通用知识**。独立于时间/空间的事实（如"法国的首都是巴黎"）。Agent 的训练语料主要属于此类 |
| 陈述性子类型 | **Episodic Memory**（情景记忆） | Declarative Memory 的子集；等价于 Experiential Memory | **自传式记忆**。包含时空上下文的特定事件（如"Agent 昨天下午与用户A讨论了天气"） |
| 功能扩展 | **Metacognitive Memory**（元认知记忆） | 独立维度（Memory about Memory） | **自反式循环**。Agent 对自身认知边界的意识（如"我知道我不具备这个信息"或评估任务复杂度） |

### 3.2 记忆类型与工程实现的映射

> ⚠️ **版本演进说明**：在图1（服务拓扑架构）中，长期记忆被划分为 **Procedural Memory** + **Semantic Memory**（两层）。而在早期 HLD 初稿中，采用的是 **Procedural Memory** + **Declarative Memory**（后者含 Semantic + Episodic 两个子类型）。版本B最终采用图1的简化模型，将 Semantic Memory 作为与 Procedural Memory 并列的顶层类别，Episodic Memory 的时序事件数据通过 Temporal KG 在 Graph 层实现，而非作为独立的记忆类型。

| **记忆类型** | **作用域** | **延迟 SLA** | **主存储引擎** | **数据结构** |
|---|---|---|---|---|
| **Working Memory** | 会话级，短暂 | < 50ms p99 | Redis Cluster 7.2+ (RedisJSON + RediSearch) | Flat KV |
| **Procedural Memory** | 长期，技能与经验 | < 200ms p95 | Neo4j + Milvus + LanceDB | Tree + Graph |
| **Semantic Memory** | 长期，知识与事实 | < 200ms p95 | Neo4j + Milvus + LanceDB | Tree + Graph |
| **Metacognitive Memory** | 跨会话，自反式 | < 200ms p95 | 附着于其他记忆类型的元数据层 | KV + Graph |

### 3.3 数据源映射规则（Data Source Mapping Rules）

以下映射规则在**管道验证时强制执行**，违规会触发管道立即拒绝并返回描述性错误码（源自补充表格2）。

| **数据源** | **目标记忆** | **允许的结构** | **设计理由** |
|---|---|---|---|
| **User-Agent Interaction** | Procedural + Declarative | **Graph ONLY** | 保留关系模式和对话流，作为可导航的图路径 |
| **Agent Execution Traces** | Procedural + Declarative | **Graph ONLY** | 建模执行轨迹和工具链为时序图路径，保持完整时空完整性 |
| **Work Documents + Source Code** | Procedural + Declarative | **Tree + Graph（Hybrid）** | 需要层级抽象用于主题归纳，AND 实体关系图用于跨文档关联（代码的 Call Graph 和 Dependency Tree） |
| **Skill Markdowns** | **Procedural ONLY** | **Tree ONLY** | 利用 Markdown 原生标题层级（H1→L1, H2→L2…）进行技能分解，最大深度 4 |

> 📌 **Working Memory 完全排除在此映射框架之外**（见第 4 章）。

#### 🔴 架构图一致性修正

在之前的分析中发现，早期 HLD 初稿的架构图中 **work documents / source code 缺少直接流向 Procedural + Declarative Memory 的箭头**，与映射规则表不一致。在本版设计中，我们明确补充了这条数据流路径：

```
Work Documents / Source Code
    ├──→ Knowledge Repository → Tree Construction（层级抽象）
    └──→ Procedural + Semantic Memory → Graph Construction（实体关系图）
```

## 4. Working Memory 架构

### 4.1 设计定位

Working Memory 是**会话级的高速上下文存储**，用于活跃的 Agent 会话。它持有当前会话的全部实时上下文，是 Agent Framework 与记忆系统交互的**第一接触点**。

这一设计借鉴了 Letta/MemGPT 的分层记忆思想——将"热数据"（当前对话上下文）与"冷数据"（长期知识）分离，通过不同的存储引擎和访问模式分别优化 [2]。

**Working Memory 包含的内容**（源自图1标注）：
- 近期消息（recent messages）
- Agent 回复（agent replies）
- 工具使用结果（tool usage results）
- 来自当前交互的 Agent 反思（agent reflections）

### 4.2 技术选型：Redis Cluster 7.2+

**选型理由**：Working Memory 的访问模式绝大多数是**点读/点写 + TTL 驱逐**，数据本质上是短暂的（会话生命周期通常 5 分钟到 2 小时）。Tree/Graph 结构在此场景下是过度设计。

#### Redis 模块矩阵

| **模块** | **关键特性** | **应用场景** |
|---|---|---|
| **RedisJSON** | 原生 JSON 数据类型，JSONPath 路径查询 | 会话状态、任务状态的结构化存储和字段级更新 |
| **RediSearch** | 全文搜索 + 向量相似度搜索 | 快速上下文关键词定位，消息模糊查询 |
| **RedisTimeSeries** | 时序数据压缩与聚合 | 工具调用延迟、Token 消耗等指标的时序记录 |
| **RedisBloom** | 布隆过滤器 | 大规模去重和存在性检测的内存优化 |

#### 基础设施规格

| **参数** | **规格** | **备注** |
|---|---|---|
| 引擎 | Redis 7.2 OSS Cluster | 6 节点集群（3 主 3 从） |
| 部署 | Kubernetes StatefulSet + 反亲和规则 | 每个主节点持有 ~5461 个哈希槽 |
| 实例类型 | r6g.2xlarge（64 GB RAM, 8 vCPU） | ARM 架构，成本优化 |
| 延迟 SLA | p50 < 10ms, p99 < 50ms | 网络 RTT 预算 < 2ms（同 AZ） |
| 吞吐量 SLA | 10K+ TPS/分片，30K+ TPS 聚合 | 启用管道批处理 |
| 驱逐策略 | volatile-ttl | 基于会话 TTL 过期 |
| 持久化 | AOF（everysec fsync）+ RDB 快照每 15 分钟 | 数据丢失窗口 ~1s |
| 最大内存 | 48 GB/节点（64 GB 的 75%） | 预留碎片化和 OS 开销余量 |

### 4.3 数据模型

| **Key 模式** | **类型** | **描述** |
|---|---|---|
| `session:{sid}:state` | Hash | 核心会话状态：agent_id, user_id, created_at, last_active_at, status |
| `session:{sid}:conv` | List（有上限） | 对话轮次，JSON 序列化消息；LTRIM 保留最近 50 轮 |
| `session:{sid}:task` | Hash | 当前任务状态：task_id, step_index, tool_chain, partial_results |
| `session:{sid}:tools` | Stream | 工具调用追踪日志；XADD + MAXLEN ~100 |
| `session:{sid}:ctx` | String (JSON) | 上下文窗口元数据：token 计数、截断游标、活跃记忆引用 |

### 4.4 TTL 策略与会话生命周期

- **默认会话 TTL**：7200 秒（2 小时），从最后一次活动计算
- 每次写入通过 **Lua 脚本原子性**更新 `last_active_at` 并重置 TTL
- **空闲会话处理**（30 分钟无活动）：状态序列化到 Kafka topic（`ams.session.archive`），然后删除键

### 4.5 会话提升管道（Session Promotion Pipeline）

当会话结束时，**提升评估器**决定是否将会话内容持久化到长期记忆：

| **触发条件** | **目标记忆** | **构建管道** |
|---|---|---|
| Agent 输出 > 500 tokens 的显著推理内容 | Semantic Memory（情景子类型） | Graph Construction |
| 工具调用追踪导致任务成功完成 | Procedural Memory | Graph Construction |
| 用户显式标记"记住这个" | Semantic Memory（语义子类型） | Tree + Graph |

提升过程是**异步的**，由会话归档 Kafka 消费者处理提取和路由。

## 5. Semantic Memory 内部结构（Tree + Graph 双轨模型）

### 5.1 设计概述

Semantic Memory 是 AMS 中信息密度最高的记忆类型，采用 **Tree + Graph 双轨存储**的混合架构。图3完整展示了其内部结构——上半部分是 Tree 层级（蓝色/绿色），下半部分是 Graph 关系网络（蓝色椭圆/橙色椭圆），两者通过 **Context Edge** 桥接。

这一双轨设计融合了两种业界前沿思路：
- **Tree 层级**：类似传统 RAG 的层级化文档分块，但增加了多级 Section 抽象
- **Graph 层级**：借鉴 LightRAG 的双层检索思想（低层级实体 + 高层级社区），但扩展为三层（Phrase Node → Community Node → Temporal KG）[3]

### 5.2 Tree 层级结构

Tree 层级实现了**渐进式层级抽象**，从原始文本块到高层语义摘要：

```
Section Level 2（顶层）
  ├── section name, summary, metadata, embedding
  ├── refer sections → [Section L1 列表]
  │
  ├── Section Level 1（中层）
  │     ├── section name, summary, metadata, embedding
  │     ├── refer blocks → [Block 列表]
  │     │
  │     ├── Block 1 (text)：raw text, tags, summary, embedding
  │     ├── Block 2 (table)
  │     ├── Block 3 (img)
  │     └── Block 4 (text)
  │
  └── Section Level 1（中层）
        └── ...
```

**关键属性**：

| **层级** | **节点属性** | **索引方式** |
|---|---|---|
| **Block**（底层） | raw text, tags, summary, embedding, source type, block name, metadata | Vector + Inverted + B-tree |
| **Section L1**（中层） | section name, summary, metadata, embedding, refer blocks | Vector + Inverted |
| **Section L2**（顶层） | section name, summary, metadata, embedding, refer sections | Vector（文档级 Cluster） |

**输出**：
- Summary → Topic / Outline / Insight（主题/大纲/洞察）
- Embedding（Summary + Critical Triples）→ Vector Cluster（仅文档级别）

**右侧标注**：Section L2 的 embedding 输出到 **cluster 1..n**，表示文档级别的向量聚类。

### 5.3 Graph 层级结构

Graph 层级实现了**语义关系网络**，从实体短语到社区摘要：

#### 节点类型

| **节点类型** | **属性** | **说明** |
|---|---|---|
| **Phrase Node**（短语节点） | name, embedding, metadata | 图的基本单元，从 Block 中抽取的实体/概念短语 |
| **Community Node**（社区节点） | summary, embedding | 通过社区检测算法聚合的高层语义单元 |

#### 边类型

| **边类型** | **属性** | **说明** |
|---|---|---|
| **Context Edge** | source_block_id | 连接 Block → Phrase Node，标记实体的来源上下文 |
| **Relation Edge** | relation_type, temporal | Phrase Node 之间的语义关系（如"调用"、"依赖"、"包含"） |
| **Synonym Edge** | confidence, temporal | Phrase Node 之间的同义/等价关系 |
| **Belongs_to Edge** | — | Phrase Node → Community Node 的归属关系 |

#### Tree ↔ Graph 的桥接

图3中最关键的设计细节是 **Context Edge**——它从 Block（Tree 层级）连接到 Phrase Node（Graph 层级），实现了两种结构的**双向可达**：

```
Block 1 (Tree) ──Context Edge──→ Phrase Node A (Graph)
Block 1 (Tree) ──Context Edge──→ Phrase Node B (Graph)
Block 4 (Tree) ──Context Edge──→ Phrase Node C (Graph)
```

这意味着：
- **从 Tree 到 Graph**：找到某个 Block 后，可以通过 Context Edge 跳转到相关的 Phrase Node，进而遍历整个关系网络
- **从 Graph 到 Tree**：找到某个 Phrase Node 后，可以通过 Context Edge 回溯到原始 Block，获取完整上下文

### 5.4 Graph Construction 的层级差异

借鉴 LightRAG 的双层检索架构 [3] 和 GraphRAG 的社区层级思想 [1]，AMS 根据数据源的不同采用不同的 Graph 构建深度：

| **来源** | **Graph 层级** | **结构** | **用途** |
|---|---|---|---|
| Procedural + Semantic Memory | **Full Three-layer** | Phrase Node → Community Node → Temporal KG | 完整的实体-关系-社区-时序图构建 |
| Knowledge Repository | **Double-layer** | Phrase Node → Community Node | 用于多跳检索（multi-hop retrieval），不含时序 |

### 5.5 遗忘机制

Tree 层级内置了两种遗忘机制，模拟人类记忆的自然衰减：

| **机制** | **说明** | **实现方式** |
|---|---|---|
| **Temporal Invalidation**（时间失效） | 超过有效期的记忆被标记为无效 | TTL 标记 + 定期清理任务 |
| **Temporal Decay**（时间衰减） | 记忆的权重随时间递减 | 指数衰减函数：$$w(t) = w_0 \cdot e^{-\lambda t}$$，其中 $$\lambda$$ 为衰减系数 |

## 6. Procedural Memory 架构

### 6.1 设计定位

Procedural Memory 存储 Agent 的**技能和经验**——"如何做"的知识。它与 Semantic Memory 共享相同的 Tree + Graph 双轨存储基础设施，但在以下方面有所不同：

| **维度** | **Semantic Memory** | **Procedural Memory** |
|---|---|---|
| **存储内容** | 事实、概念、实体关系 | 工作流、决策模式、工具使用配方 |
| **Graph 节点语义** | 实体短语（名词性） | 动作步骤（动词性） |
| **Graph 边语义** | 语义关系（is-a, has-a, related-to） | 执行序列（then, if-then, depends-on） |
| **时序重要性** | 中等（事实可能更新） | 高（执行顺序是核心） |
| **主要数据源** | 文档、交互中的事实性内容 | 执行追踪、成功的工具调用链、skill.md |

### 6.2 Skill 生命周期

```
Agent Execution Trace
    ↓ (Pattern Detection)
Procedural Memory (Graph: 执行轨迹图)
    ↓ (Skill Mining Pipeline)
Skill Candidate
    ↓ (Sandbox Testing)
Validated Skill
    ↓ (Codification)
skill.md → Skill-MDS → Agent Framework
    ↓ (New Execution)
New Trace → ... (循环)
```

## 7. Memory Processing Pipeline

### 7.1 管道总览

Memory Processing Pipeline 是连接数据源与记忆存储的**核心处理引擎**，负责数据的清洗、转换、结构化和索引。

```
┌──────────────────────────────────────────────────────────────┐
│                   Memory Processing Pipeline                  │
│                                                              │
│  ┌─────────┐   ┌──────────┐   ┌───────────┐   ┌──────────┐ │
│  │ Ingestion│→ │ Transform │→ │ Structure  │→ │ Indexing  │ │
│  │  Stage   │   │  Stage   │   │   Stage    │   │  Stage   │ │
│  └─────────┘   └──────────┘   └───────────┘   └──────────┘ │
│                                                              │
│  Kafka Topics:                                               │
│  ams.raw.* → ams.transformed.* → ams.structured.* → Storage │
└──────────────────────────────────────────────────────────────┘
```

### 7.2 四阶段处理流程

| **阶段** | **输入** | **处理内容** | **输出** |
|---|---|---|---|
| **① Ingestion** | 原始数据（Trace/文档/代码） | 格式检测、数据源分类、映射规则验证 | 标准化的原始记录 |
| **② Transform** | 标准化记录 | 文本分块、摘要生成、实体抽取、三元组提取 | 结构化的中间表示 |
| **③ Structure** | 中间表示 | Tree Construction（Block→Section→L2）+ Graph Construction（Node→Edge→Community） | Tree 节点 + Graph 节点/边 |
| **④ Indexing** | 结构化节点 | Embedding 生成、向量索引、倒排索引、B-tree 索引、图索引 | 可检索的索引条目 |

### 7.3 Tree Construction Pipeline

**输入**：Raw Block（text / code / img / table）

**处理流程**：

1. **Block 解析**：将原始文档拆分为 Block，每个 Block 包含 raw text, tags, summary, embedding, source type, block name, metadata
2. **Section L1 归属**：基于文档结构（标题层级、段落边界）将 Block 归属到 Section L1
3. **Section L2 抽象**：将多个 Section L1 聚合为 Section L2，生成更高层的摘要
4. **Summary/Embedding 生成**：为每个层级生成摘要文本和向量嵌入
5. **索引构建**：Vector / Inverted / B-tree 三种索引

**Block 间链接**：同一文档/会话内的 Block 通过 prev/next 链接形成有序序列。

### 7.4 Graph Construction Pipeline

**输入**：Block 的结构化内容（已完成 Tree Construction）

**处理流程**：

1. **Phrase Node 抽取**：从 Block 中提取实体/概念短语作为图节点
2. **Context Edge 构建**：连接 Block → Phrase Node（Tree↔Graph 桥接）
3. **Relation/Synonym Edge 构建**：通过 LLM 或规则引擎识别节点间的语义关系
4. **Community 检测**：使用 Leiden 算法进行社区检测，生成 Community Node 及其 Summary
5. **Temporal KG 生成**（仅 full three-layer）：为时序敏感的关系添加时间戳和有效期

**与 LightRAG 的对比**：LightRAG 采用"去重 + 增量更新"的轻量级图构建策略 [4]，AMS 在此基础上增加了 Community 层级和 Temporal KG，以支持更复杂的多跳推理和时序查询。

## 8. Storage & Retrieval 架构

### 8.1 存储引擎矩阵

| **引擎** | **用途** | **存储内容** |
|---|---|---|
| **Redis Cluster** | Working Memory | 会话状态、对话历史、任务上下文 |
| **Neo4j** | Graph Storage | Phrase Node、Edge、Community Node、Temporal KG |
| **Milvus** | Vector Storage | Block/Section/Node 的 Embedding 向量 |
| **LanceDB** | Hybrid Storage | 向量 + 元数据的列式存储，支持高效过滤 |
| **Elasticsearch** | Full-text Index | 倒排索引、BM25 全文检索 |
| **PostgreSQL** | Metadata Storage | 系统配置、用户偏好、审计日志 |

### 8.2 检索架构

检索服务是 AMS 的**核心价值输出点**，提供两种检索模式：

#### Lightweight Mode（轻量模式）

适用于简单查询，延迟优先：

```
User Query
    ├── BM25 Search (Full-text) ── Elasticsearch
    ├── Vector Search ── Milvus
    ├── Tree Search ── Neo4j (层级导航)
    └── Graph Search + PPR ── Neo4j (个性化 PageRank)
    ↓
RRF Fusion（倒数排名融合）
    ↓
Results
```

#### Agentic Mode（智能体模式）

适用于复杂查询，质量优先：

```
User Query
    ↓
Query Expansion (LLM) ── 查询扩展/改写
    ↓
Multi-round Retrieval ── 多轮迭代检索
    ↓
Intelligent Fusion (LLM) ── 智能融合/推理
    ↓
Reranking (Optional)
    ↓
Results
```

这一双模式设计与 AgenticRAG 的理念一致——简单查询走确定性管道以保证延迟，复杂查询走 LLM 驱动的智能管道以保证质量 [1]。

### 8.3 Structured Memory Retrieval（结构化记忆检索）

图2右侧标注了检索服务的完整能力：

| **能力** | **说明** |
|---|---|
| 语义检索 | 基于向量相似度的语义匹配 |
| 精准匹配 | 基于关键词/ID 的精确查找 |
| 粗召回 | 多路召回（BM25 + Vector + Graph + Tree） |
| 精排 | Cross-encoder Reranking 或 LLM Reranking |
| 结构化拼装 | 将检索结果组装为 Agent 可消费的结构化上下文 |

## 10. Multi-Agent 协作

### 10.1 记忆共享模型

在多 Agent 场景下，AMS 支持三种记忆共享模型：

| **模型** | **说明** | **适用场景** |
|---|---|---|
| **Private Memory** | 每个 Agent 拥有独立的记忆空间 | 隐私敏感任务 |
| **Shared Memory** | 多个 Agent 共享同一记忆空间 | 协作任务（如多 Agent 研究） |
| **Hierarchical Memory** | 主 Agent 拥有全局记忆，子 Agent 拥有局部记忆 + 只读访问全局记忆 | 编排型多 Agent 系统 |

### 10.2 一致性保证

| **记忆类型** | **一致性模型** | **实现方式** |
|---|---|---|
| Working Memory | 强一致性（同一会话内） | Redis 单分片内的原子操作 |
| Long-term Memory | 最终一致性 | Kafka 保序 + 异步写入 + 版本向量 |

## 15. 关键设计决策记录（ADR）

| **ADR#** | **决策** | **理由** | **替代方案** |
|---|---|---|---|
| ADR-001 | Working Memory 使用 Redis 而非 Tree/Graph | 会话级数据的访问模式是点读/点写+TTL，Tree/Graph 是过度设计 | SQLite、Memcached |
| ADR-002 | 长期记忆采用 Tree + Graph 双轨存储 | 模拟人类认知的两种访问模式：层级导航 + 关联跳转 | 仅 Graph、仅 Vector |
| ADR-003 | Graph Construction 区分 full three-layer 和 double-layer | 静态文档不需要时序图，减少构建开销 | 统一使用 three-layer |
| ADR-004 | 检索服务提供 Lightweight + Agentic 双模式 | 简单查询走确定性管道保延迟，复杂查询走 LLM 管道保质量 | 统一使用 Agentic 模式 |
| ADR-005 | 遥测采集采用 Sidecar 模式而非 SDK 嵌入 | 零代码改动、语言无关、可独立升级 | Agent SDK 内嵌 |
| ADR-006 | 会话提升采用异步 Kafka 管道而非同步写入 | 避免会话结束时的延迟尖峰，提升用户体验 | 同步双写 |
| ADR-007 | Semantic Memory 与 Procedural Memory 共享存储引擎 | 减少运维复杂度，通过 namespace/label 隔离 | 独立存储集群 |
