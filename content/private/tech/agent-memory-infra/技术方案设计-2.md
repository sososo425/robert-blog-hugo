| --------------------- | ----------------------------- | --------------------------------------- |
| **记忆演化机制缺失**          | 只设计了"存"和"查"，没有"如何随时间演化"       | 补充 Reflection / Consolidation 机制        |

| **记忆遗忘与压缩**           | 没有 TTL 策略以外的遗忘/压缩机制           | 引入基于重要性的遗忘、定期 Consolidation             |
| **Multi-Agent 记忆共享**  | 架构中提到了 Multi-Agent 但没有说如何共享记忆 | 增加 Shared Memory / Memory Router 设计     |
| **技能挖掘管道细节不足**        | Skill-MDS 概念清晰但工程实现缺失         | 补充技能自动评估和版本迭代机制                         |
| **自主记忆管理**            | 记忆操作完全被动，靠外部系统触发              | 引入 Agent 自主记忆管理（MemGPT 思路）              |

## 第一章：整体方案概述

### 1.1 设计目标

构建一套**生产级 Agent 认知基础设施（Agent Cognitive Infrastructure）**，使 AI Agent 具备：

| 能力维度     | 目标                  | 对应组件                             |
| -------- | ------------------- | -------------------------------- |
| **即时感知** | 当前任务的精准上下文管理        | Working Memory + Context Manager |
| **经验积累** | 从每次执行中自动提取和沉淀技能     | Episode Store + Skill-MDS        |
| **知识检索** | 多粒度、多路径的语义检索        | Semantic Memory（GraphRAG 架构）     |
| **记忆演化** | 记忆自动整理、抽象、压缩、遗忘     | Memory Processing Pipeline       |
| **知识传承** | 经验跨 Agent 复用，团队智能积累 | Shared Procedural Memory         |
| **可追溯性** | 每个决策都有完整证据链         | Agent-TES + Chain of Evidence    |

### 1.2 核心设计原则

**原则 1：认知分层（Cognitive Layering）**

参照认知科学的记忆分层理论（Tulving 的情景/语义二分法 + Baddeley 工作记忆模型 + MemGPT 的 OS 隐喻），将记忆分为四层：

```
┌──────────────────────────────────────────────────┐
│  L1: Working Memory（工作记忆）                    │
│     当前推理上下文，时效秒~分钟，容量极有限          │
├──────────────────────────────────────────────────┤
│  L2: Episode Memory（情景记忆）                    │
│     经历过的具体事件流，时效天~月，支持时序查询       │
├──────────────────────────────────────────────────┤
│  L3: Procedural Memory（程序性记忆）               │
│     从 Episode 归纳出的可复用技能，永久              │
├──────────────────────────────────────────────────┤
│  L4: Semantic Memory（语义记忆）                   │
│     事实性知识库，多粒度 GraphRAG 结构，永久          │
└──────────────────────────────────────────────────┘
```

> v1 中把 Episodic Memory 和 Semantic Memory 混为一体，v2 明确分离。这是本架构最重要的改动之一。

**原则 2：写入驱动（Write-First）**

记忆的真正价值在于"写入时的质量"，而非检索算法。高质量的写入管道（包含实体抽取、摘要生成、关系推理）决定了检索的上限。

**原则 3：记忆演化（Memory Evolution）**

参照 Generative Agents 的 Reflection 机制和 MemOS 的分层管理思路：

- **实时层**：原始事件写入（高频，低延迟）
- **定时层**：Consolidation 任务（低频，高成本 LLM 调用）
- **触发层**：达到阈值时（如重要性分数高、失败次数多）自动触发反思

**原则 4：时间感知（Temporal Awareness）**

引入 Zep/Graphiti 的双时间模型：

- **Event Time**：事件实际发生时间（用于时序推理）
- **Ingestion Time**：数据写入系统时间（用于审计和版本控制）

**原则 5：Agent 自主管理（Agentic Memory Management）**

参照 MemGPT/Letta 的设计，Agent 能够：
- 主动决定将哪些内容固化到长期记忆
- 主动触发自身的记忆检索
- 感知上下文压力，自动触发换页（Swap）

### 2.2 Agent Framework（执行中枢）

#### 2.2.1 Context Manager（上下文管理器）

这是 v2 相对 v1 新增的核心组件，直接来源于 MemGPT 的 OS 隐喻：将 LLM 上下文窗口视为"有限 RAM"，Context Manager 负责管理"换页"策略。

```
┌──────────────────────────────────────────────────────┐
│               Main Context (LLM 上下文窗口)           │
│  ┌────────────┐  ┌────────────┐  ┌────────────────┐  │
│  │  System    │  │  Working   │  │   FIFO         │  │
│  │ Instructions│  │  Context  │  │   History      │  │
│  │ (固定指令)  │  │(当前任务状态)│  │  (最近N轮对话) │  │
│  └────────────┘  └────────────┘  └────────────────┘  │
│                                                      │
│  触发条件：context_usage > 70% → summarize           │
│            context_usage > 95% → evict + archive    │
└──────────────────────────────────────────────────────┘
              ↑↓ Memory Tool Calls
┌─────────────────────────────────────────────────────┐
│              Agentic Memory（外部存储）              │
│   L2 Episode / L3 Procedural / L4 Semantic          │
└─────────────────────────────────────────────────────┘
```

**Agent 可主动调用的 Memory 原语**：

| 原语 | 说明 | 触发场景 |
|------|------|---------|
| `memory.recall(query)` | 检索外部记忆注入上下文 | 任务开始时，需要相关经验 |
| `memory.store(content, type)` | 将重要内容持久化 | 发现值得记录的结论 |
| `memory.reflect()` | 触发对当前 Episode 的反思 | 任务完成后，主动抽象 |
| `memory.archive(msg_ids)` | 压缩旧消息到 Episode Store | 上下文即将超限 |

#### 2.2.2 技术选型

Agent Framework 采用 **LangGraph** 深度定制：

- LangGraph 的 `StateGraph` 对应 Working Memory 的状态机
- LangGraph 的 `Checkpointer` 对应 Episode Memory 的自动保存
- LangGraph 的 `Store` 对应 Procedural/Semantic Memory 的检索接口

### 2.3 Agent-TES（遥测与证据服务）

#### 2.3.1 设计目标

非侵入式全链路观测，是 Memory Processing Pipeline 的原始数据来源。

#### 2.3.2 三层采集架构

| 层 | 技术 | 采集内容 | 侵入性 |
|----|------|---------|--------|
| **基础设施层** | eBPF + Sidecar | 系统调用、服务间网络 | 零侵入 |
| **应用层** | SDK 埋点 | 业务语义（意图、实体、结果） | 点状侵入 |
| **汇聚层** | Kafka / Pulsar | 流式传输到 Memory Pipeline | 无 |

#### 2.3.3 Evidence Schema（证据数据结构）

每条采集记录是一个**结构化 Evidence**，包含以下核心字段：

```json
{
  "evidence_id": "evid_abc123",
  "trace_id": "trace_xyz",
  "span_id": "span_001",

  "event_time": "2026-03-20T10:00:00.000Z",
  "ingestion_time": "2026-03-20T10:00:00.050Z",

  "agent_id": "agent_cs_v2",
  "session_id": "sess_user123",
  "user_id": "user_123",

  "evidence_type": "LLM_CoT | TOOL_CALL | USER_FEEDBACK | PLAN | OUTCOME",

  "content": {
    "input": "...",
    "output": "...",
    "thought_process": "...",
    "confidence": 0.92
  },

  "outcome": {
    "status": "success | failure | partial",
    "user_feedback": "positive | negative | neutral",
    "resolution_time_ms": 2300
  },

  "importance_score": 7.5
}
```

> `importance_score` 是 v2 新增字段，借鉴 Generative Agents 的"重要性打分"机制：由 LLM 对每条 Evidence 打 1-10 分，用于后续记忆检索排序和遗忘策略。

#### 3.2.1 Episode Processing 管道详细流程

```
原始 Evidence
    ↓
1. 写入 Episode Store（LanceDB，append-only）
    ↓
2. 重要性评分（LLM 异步打分，1-10 分）
    ↓
3. 实体提取（快速 NER，写入实体缓存）
    ↓
4. 触发判断：
   │  - importance >= 8 → 立即触发 Reflection
   │  - episode_count % 100 == 0 → 批量 Consolidation
   │  - session_end → 触发 Session Summary
   ↓
5. Reflection（LLM 批量处理，异步）
   │  输入：最近 N 条高重要性 Episodes
   │  输出：高阶抽象洞察（写入 Semantic Memory）
   ↓
6. Skill Candidate 生成（识别潜在可复用模式）
```

#### 3.2.2 Consolidation（记忆整合）机制

这是 v2 新增的关键机制，参照 Generative Agents 的 Reflection + MemOS 的分层整理：

```python
class MemoryConsolidator:
    """
    定期将 Episode Memory 中的低层次经历
    抽象为 Semantic Memory 中的高层次知识
    """

    def consolidate(self, agent_id: str, time_window_hours: int = 24):
        # 1. 取出时间窗口内的高重要性 Episodes
        episodes = self.episode_store.query(
            agent_id=agent_id,
            min_importance=6.0,
            hours_ago=time_window_hours
        )

        if len(episodes) < self.min_episodes_threshold:
            return  # 不够多，不触发

        # 2. 用 LLM 做"反思"
        reflection_prompt = f"""
        以下是 Agent {agent_id} 最近 {time_window_hours} 小时内的关键经历：

        {self._format_episodes(episodes)}

        请从中提炼：
        1. 发现的规律或洞察（高阶知识）
        2. 反复出现的成功模式（Skill 候选）
        3. 反复出现的失败原因（需要规避的陷阱）
        4. 对外部世界的新认知（Factual Memory 更新）

        以结构化 JSON 格式输出。
        """

        reflection = await self.llm.generate(reflection_prompt)

        # 3. 将反思结果写入不同 Memory 层
        for insight in reflection.get("insights", []):
            await self.semantic_memory.upsert_knowledge(insight)

        for skill_candidate in reflection.get("skill_candidates", []):
            await self.skill_mds.submit_candidate(skill_candidate)

        # 4. 标记已整合的 Episodes
        await self.episode_store.mark_consolidated(
            episode_ids=[e.id for e in episodes]
        )
```

## 第四章：四层记忆系统详细设计

### 4.1 L1: Working Memory（工作记忆）

**定位**：当前任务的活跃上下文，时效秒~分钟级，容量极有限。

**与 v1 的差异**：v2 增加了 Context Manager 的换页逻辑，以及 Agent 可主动调用的 Memory 原语（recall/store/archive）。

**数据结构**：

```json
{
  "wm_id": "wm_sess_abc123",
  "agent_id": "agent_cs_v2",
  "session_id": "sess_user123",

  "active_goal": {
    "primary": "处理用户退货请求",
    "sub_goals": ["验证订单", "检查政策", "生成退货单"]
  },

  "retrieved_memories": [
    {
      "source": "semantic",
      "memory_id": "blk_return_policy",
      "content": "30天无理由退货...",
      "relevance": 0.95
    },
    {
      "source": "procedural",
      "skill_id": "skill_return_handling_v3",
      "success_rate": 0.87
    }
  ],

  "current_entities": {
    "order_id": "ORD-001",
    "user_tier": "VIP",
    "intent": "return_request"
  },

  "context_usage_ratio": 0.62,
  "ttl_seconds": 3600
}
```

**存储**：Redis，亚毫秒延迟 + 原生 TTL。

**换页策略**：

```
context_usage < 70%  → 正常运行
context_usage ≥ 70%  → 压缩最旧的对话历史
context_usage ≥ 95%  → 将 FIFO 队列中最旧的消息 archive 到 Episode Store
```

### 4.2 L2: Episode Memory（情景记忆）

> **这是 v2 最重要的新增层**，v1 中没有独立建模。

**定位**：Agent 经历过的具体事件流（类比人类的"我昨天做了什么"），是 Semantic Memory 的原料来源，也是技能挖掘的数据基础。

**设计来源**：
- Generative Agents 的 Memory Stream（事件流存储 + Reflection）
- Zep/Graphiti 的 Episode Subgraph（带时间戳的原始对话和行为）
- MemRL 的 Episodic Memory（运行时强化学习的经验回放源）

**与 Semantic Memory 的区别**：

| 维度 | Episode Memory | Semantic Memory |
|------|---------------|----------------|
| **内容性质** | "发生了什么"（具体事件） | "是什么"（抽象知识） |
| **时序** | 强时序，append-only | 弱时序，可更新 |
| **粒度** | 事件级（单次对话/执行） | 知识级（提炼后的洞察）|
| **来源** | 直接来自执行 | 由 Episode 整合而来 |
| **查询方式** | 时序 + 相似度 | 语义检索 + 图遍历 |

**Episode 数据结构**：

```json
{
  "episode_id": "ep_20260320_abc123",
  "agent_id": "agent_cs_v2",
  "session_id": "sess_user123",
  "user_id": "user_456",

  "event_time": "2026-03-20T10:00:00Z",
  "ingestion_time": "2026-03-20T10:00:01Z",

  "episode_type": "task_execution | reflection | plan | observation",
```

```python
class SkillHealthMonitor:
    """
    监控技能质量，自动触发更新或废弃
    """
    def check_degradation(self, skill_id: str) -> SkillHealthStatus:
        skill = self.skill_store.get(skill_id)
        recent_applications = self.get_recent_applications(
            skill_id, days=7
        )

        if len(recent_applications) < 10:
            return SkillHealthStatus.INSUFFICIENT_DATA

        recent_success_rate = self._calc_success_rate(recent_applications)
        delta = skill.baseline_success_rate - recent_success_rate

        if delta > 0.15:  # 成功率下降超过 15%
            self.alert_skill_degradation(skill_id, delta)
            self.trigger_skill_refinement(skill_id)
            return SkillHealthStatus.DEGRADED

        if skill.consecutive_failures >= 5:
            self.deprecate_skill(skill_id)
            return SkillHealthStatus.DEPRECATED

        return SkillHealthStatus.HEALTHY
```

### 4.4 L4: Semantic Memory（语义记忆）

> **对应架构图3**：Semantic Memory 内部存储结构

#### 4.4.1 五层分层存储体系

这是整个架构的"事实知识库"，采用 GraphRAG 架构，从下到上共五层：

```
┌─────────────────────────────────────────────────────────┐
│  L5: Community Summary（社区级摘要）                     │
│      跨知识领域的主题综合理解                             │
│      查询：宏观综合性问题                                 │
├─────────────────────────────────────────────────────────┤
│  L4: Section Summary（章节级摘要）                       │
│      文档章节级别的语义摘要                               │
│      查询：某个主题的要点                                 │
├─────────────────────────────────────────────────────────┤
│  L3: Content Block（内容块，RAG 层）               ◀ 核心 │
│      实际内容的切分存储，传统 RAG 的粒度                   │
│      查询：具体问题的精确答案                             │
├─────────────────────────────────────────────────────────┤
│  L2: Phrase Node + Edge（实体-关系图谱）                  │
│      从 Block 中提取的实体节点及其关联关系                 │
│      查询：关系推理、多跳关联                             │
├─────────────────────────────────────────────────────────┤
│  L1: Community Node（知识社区）                          │
│      通过 Leiden 算法聚类的语义相关实体集合               │
│      为 L5 Community Summary 提供结构基础                │
└─────────────────────────────────────────────────────────┘
```

#### 4.4.2 你对图3的解读补充与修正

关于图3的五层，你的理解基本正确，这里做精确描述：

- **第3层（绿色，Block 层）**：就是传统 RAG 的切块层，是整个结构的"真实内容"基础。每个 Block 对应一段原始文本，有独立的 Embedding 向量用于语义检索。

- **Block 上层（Section Summary L1/L2）**：对多个 Block 进行 LLM 摘要，层次越高，抽象程度越高。Section L1 是章节摘要，Section L2 是文档摘要。它们有自己的 Embedding，用于"宏观主题检索"（当用户问的是全局性问题时用这条路径）。

- **Block 下层（Phrase Node + Edge）**：这就是"实体 Node 提取"。从每个 Block 的文本中用 NER + 关系抽取提取出实体节点，Block 和 Node 之间有 `CONTAINS` 边，Node 和 Node 之间有 `RELATION` 边（语义关系）。

- **最下层（Community Node Summary）**：这是你说"没看明白"的部分。

  **详细解释**：在由 Phrase Node 构成的知识图谱中，有些节点之间内部连接很密集（互相引用、共同出现频繁），而与外部节点连接稀疏——这一批节点就形成了一个"社区（Community）"。类比现实世界：退货、退款、售后、商品完好、30天……这些概念频繁一起出现，它们就聚成"售后服务"社区。

  Leiden 算法自动检测这些社区，然后用 LLM 对每个社区生成一段综合摘要（Community Summary）。这个摘要的价值在于：它能回答"给我总结一下退货相关的所有知识"这类**跨 Block 的宏观问题**，而传统 RAG 对这类问题完全无能为力（只能返回零散片段）。

  Community 也有层次结构：Level 0 社区聚合为 Level 1 社区，Level 1 再聚合为 Level 2——越高层越抽象，覆盖范围越大。

## 第五章：Skill-MDS（技能挖掘与发现服务）

### 5.1 设计目标

从 Episode Memory 中**自动挖掘、评估、管理**可复用技能，是连接"经验"和"能力"的桥梁。

### 5.2 技能挖掘完整流程

```
输入：高质量 Episode（success=true, importance>=7）
    ↓
Step 1: Episode Clustering（相似场景聚类）
    基于任务 Embedding 的 K-Means 或 HDBSCAN 聚类
    同一 Cluster 的 Episodes 具有相似的任务背景
    ↓
Step 2: Action Pattern Extraction（动作序列提取）
    使用 PrefixSpan 或 GSP 序列模式挖掘算法
    找出 Cluster 内高频出现的动作子序列
    ↓
Step 3: Variability Analysis（变化性分析）
    区分固定步骤（所有 Episode 都有）和可变步骤（部分 Episode 有）
    可变步骤提炼为参数或分支条件
    ↓
Step 4: LLM Skill Template Generation（技能模板生成）
    输入：提取出的动作模式
    输出：结构化的 Skill JSON（preconditions + workflow）
    ↓
Step 5: Quality Gate（质量门控）
    计算候选技能在源 Episodes 上的"回放成功率"
    success_rate < 0.75 → 拒绝，不生成 Skill
    ↓
Step 6: Deduplication（去重与合并）
    与现有 Skill Library 对比
    相似度 > 0.9 → 考虑合并或版本迭代，而非新建
    ↓
输出：新 Skill 或 Skill 版本升级
```

### 5.3 Skill Mesh（技能关系图）

```
技能之间的关系类型：

DEPENDS_ON      → skill A 的执行需要先执行 skill B
OFTEN_COMBINED  → skill A 和 B 经常被一起使用（共现分析）
SUPERSEDES      → skill A 是 skill B 的更好替代
SPECIALIZES     → skill A 是 skill B 在特定场景下的专化版本
CONFLICTS_WITH  → skill A 和 B 不能同时激活
```

这个关系图用于：
1. **Skill 推荐**：激活 skill A 时，自动推荐 `OFTEN_COMBINED` 的 skill B
2. **Skill 清理**：当 skill A `SUPERSEDES` skill B 时，自动废弃 B
3. **执行规划**：`DEPENDS_ON` 关系用于自动排序 skill 的执行顺序

### 6.1 四层记忆的检索路径

```
用户 Query
    │
    ▼ 意图分析
    │  - 局部精确？ → 优先 Block Vector Search
    │  - 宏观概览？ → 优先 Community Summary

    │

    │     工具：LanceDB ANN，top-k=10
    │
    ├── 路径2: Section Summary Search（主题匹配）
    │     工具：LanceDB / PostgreSQL，top-k=3
    │
    ├── 路径3: Community Summary（全局理解）
    │     工具：PostgreSQL pgvector，top-k=2
    │
    ├── 路径4: Graph Traversal（关系推理）
    │     工具：Neo4j Cypher，max_hops=3
    │
    ├── 路径5: Skill Retrieval（技能匹配）
    │     工具：pgvector + 条件过滤
    │
    └── 路径6: Episode Retrieval（历史经历）
          工具：LanceDB，时序 + 语义双过滤
    │
    ▼ 重排序（Reranking）
    │  Cross-Encoder 对候选结果重排
    │  + 混合评分（相关性 × 0.6 + 新近性 × 0.3 + 重要性 × 0.1）
    │
    ▼ Context Assembly
       按逻辑顺序组装，附带引用溯源信息
```

### 6.2 混合评分算法

参照 Generative Agents 的三维评分机制：

```python
def calculate_memory_score(memory: Memory, query: Query) -> float:
    """
    综合评分 = 相关性 + 新近性 + 重要性
    """
    # 1. 相关性（语义相似度）
    relevance = cosine_similarity(query.embedding, memory.embedding)

    # 2. 新近性（指数衰减）
    hours_ago = (now() - memory.event_time).total_hours()
    recency = math.exp(-hours_ago / 24)  # 24h 半衰期

    # 3. 重要性（LLM 预打分，1-10，归一化到 0-1）
    importance = (memory.importance_score - 1) / 9

    # 加权求和
    return 0.6 * relevance + 0.3 * recency + 0.1 * importance
```

### 6.3 GraphRAG 检索模式

针对不同问题类型，选择不同的 GraphRAG 检索模式（借鉴 Microsoft GraphRAG 论文）：

| 问题类型 | 检索模式 | 说明 |
|---------|---------|------|
| **局部精确**（"退货需要几天？"）| Local Search | Block 向量检索 + 相邻图扩展 |
| **全局概览**（"售后政策有哪些内容？"）| Global Search | Community Summary 层级检索 |
| **关系推理**（"A 和 B 有什么关联？"）| Graph Traversal | Cypher 多跳查询 |
| **时序查询**（"上次出现这个问题是什么时候？"）| Temporal Search | Episode Store 时序查询 |
| **技能匹配**（"处理这类问题用什么方法？"）| Skill Retrieval | pgvector + 条件过滤 |

```cypher
    FOR (b:Block) REQUIRE b.block_id IS UNIQUE;
CREATE CONSTRAINT community_id IF NOT EXISTS
    FOR (c:Community) REQUIRE c.community_id IS UNIQUE;

// === 索引 ===
CREATE INDEX phrase_text IF NOT EXISTS FOR (p:Phrase) ON (p.text);
CREATE INDEX phrase_community IF NOT EXISTS FOR (p:Phrase) ON (p.community_id);
CREATE INDEX block_source IF NOT EXISTS FOR (b:Block) ON (b.doc_id);
CREATE INDEX block_event_time IF NOT EXISTS FOR (b:Block) ON (b.event_time);

// === 边类型 ===
// (Block)-[:CONTAINS {confidence, positions}]->(Phrase)
// (Phrase)-[:RELATES_TO {relation_type, confidence}]->(Phrase)
// (Phrase)-[:SYNONYM_OF]->(Phrase)
// (Phrase)-[:PART_OF]->(Phrase)          # 上下位关系
// (Phrase)-[:BELONGS_TO]->(Community)
// (Community)-[:CONTAINS]->(Community)  # 社区层次

// === 示例创建 ===
CREATE (p:Phrase {
    phrase_id:      'phrase_退货政策_001',
    text:           '退货政策',
    phrase_type:    'CONCEPT',
    canonical_form: '退货政策',
    aliases:        ['退款政策', '退换货规定'],
    mention_count:  23,
    community_id:   'community_after_sales_l0',
    event_time:     datetime('2024-01-15'),
    ingestion_time: datetime('2026-03-19')
});
```

### 7.4 LanceDB Schema（Episode Store + Multimodal Lake）

```python
import lancedb
import pyarrow as pa
from lancedb.pydantic import LanceModel, Vector

class EpisodeRecord(LanceModel):
    episode_id:     str
    agent_id:       str
    session_id:     str
    user_id:        str

    event_time:     pa.timestamp('us')
    ingestion_time: pa.timestamp('us')

    episode_type:   str    # task_execution | reflection | plan
    raw_content:    str    # JSON
    summary:        str | None

    importance_score: float  # 1-10
    consolidated:   bool
    outcome:        str    # success | failure | partial

    embedding:      Vector(1536)   # 基于 summary

    entities:       str    # JSON array
    skills_applied: str    # JSON array


class ContentBlock(LanceModel):
    block_id:       str
    content_type:   str    # paragraph | table | image_caption | code
    source_type:    str    # business_doc | agent_execution

    raw_text:       str | None
    raw_binary:     bytes | None
    metadata:       str    # JSON

    embedding:      Vector(1536)

    event_time:     pa.timestamp('us')
    ingestion_time: pa.timestamp('us')

    parent_section_l1: str | None
    parent_section_l2: str | None
    doc_id:         str | None

    quality_score:  float
    access_count:   int = 0


# 创建连接并建表
db = lancedb.connect("s3://your-bucket/memory-lake")

episode_table = db.create_table("episodes", schema=EpisodeRecord.to_arrow_schema())
block_table   = db.create_table("content_blocks", schema=ContentBlock.to_arrow_schema())

# 创建向量索引
episode_table.create_index(
    metric="cosine",
    num_partitions=128,
    num_sub_vectors=16,
    vector_column_name="embedding"
)
block_table.create_index(
    metric="cosine",
    num_partitions=256,
    num_sub_vectors=16,
    vector_column_name="embedding"
)
```

## 第九章：Multi-Agent 记忆共享（v2 新增）

### 9.1 共享记忆架构

多 Agent 场景中，记忆需要在不同维度共享：

```
Agent A (个人助手)     Agent B (代码 Agent)    Agent C (搜索 Agent)
    │                       │                       │
    └───────────────────────┼───────────────────────┘
                            │
              ┌─────────────▼───────────────┐
              │      Memory Router           │
              │                              │
              │  路由规则：                  │
              │  • 按 domain 路由             │
              │  • 按 agent_type 路由         │
              │  • 按 visibility 控制访问     │
              └───────────────────────────── ┘
                            │
         ┌──────────────────┼────────────────────┐
         ▼                  ▼                    ▼
  ┌────────────┐    ┌──────────────┐    ┌─────────────┐
  │ Private    │    │   Shared     │    │  Public     │
  │  Memory    │    │   Memory     │    │  Knowledge  │
  │（Agent私有） │    │（团队共享）   │    │（全局知识库） │
  └────────────┘    └──────────────┘    └─────────────┘
```

### 9.2 记忆可见性模型

| 可见性级别 | 说明 | 适用内容 |
|----------|------|---------|
| `private` | 仅该 Agent 可访问 | 个性化偏好、私有推理过程 |
| `team` | 同组 Agent 共享 | 部门级技能、团队知识 |
| `org` | 组织内全部 Agent | 公司知识库、通用政策 |
| `public` | 对外暴露（如知识产品） | 产品文档、FAQ |

## 第十一章：v2 vs v1 对比总结

| 维度 | v1 | v2 |
|------|----|----|
| **记忆分层** | 3层（Working/Procedural/Semantic）| 4层（+Episode 独立层）|
| **时间感知** | 简单 TTL | 双时间模型（EventTime + IngestionTime）|
| **记忆演化** | 无 | Reflection + Consolidation 机制 |
| **写入管道** | 模糊提及 | 独立设计，多 Stage 流水线 |
| **Agent 自主性** | 被动接受检索 | 可主动 recall/store/reflect/archive |
| **技能管理** | 无版本历史 | 版本历史 + 退化检测 + Skill Mesh |
| **Multi-Agent** | 提及未设计 | Memory Router + 可见性模型 |
| **社区检测** | 有设计，缺解释 | 详细说明 Leiden + 分层 Community |
| **重要性打分** | 未引入 | LLM 打分（1-10），影响检索排序和遗忘策略 |
